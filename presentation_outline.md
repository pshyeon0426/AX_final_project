## (표지) 슈퍼SOL 고객 반응 분석 · 경쟁 벤치마킹 · 리뷰 답글 초안
4조(댓글부대) · 신한은행 AX 전문가 Intensive 최종 프로젝트

## 1. 문제 정의 & 금융 업무 활용 가능성
- **문제**: 출시 초기 고객 반응을 빠르게·일관되게 파악하기 어렵고, 경쟁 비교/답글 응대가 수작업
- **활용**: VOC 모니터링 자동화, 경쟁 포지셔닝, CX 답글 생산성, 개선 우선순위 도출
- 핵심 메시지: **공개 리뷰만으로** 출시 초기 CX 의사결정 지원 (내부 데이터 불필요)

## 2. 데이터 수집 & EDA/전처리 판단 근거
- 수집: Google Play(google-play-scraper) + App Store(**iTunes 공개 RSS**) + 수동/샘플, 근거는 `source_manifest.csv`
- 대상: 슈퍼SOL + 경쟁 5개(KB·우리·하나·카카오뱅크·토스), ID는 검색 API로 검증
- 전처리 판단: 결측·중복·초단문 제거, **PII 마스킹**([MASKED]), 날짜 표준화, 길이 IQR 이상치
- 결과: 원본 ~3,400 → 정제 ~2,976건 (그래프: 단계별 제거 건수)

## 3. 시스템 아키텍처
- 파이프라인: 수집 → 전처리 → 검증 → (감성/이슈/토픽/벤치마킹) → 리포트 → 답글 → 평가
- 단일 오케스트레이션 `pipeline.py`, 표현 계층 `app.py`(Streamlit), 공용 테마 `theme.py`
- 다이어그램 1장(README §아키텍처 흐름 활용)

## 4. ML baseline 대비 개선 결과
- 라벨: **별점 약지도**(1~2 neg / 3 neu / 4~5 pos) — *실제 감정 정답 아님* 명시
- Dummy 0.23 → baseline(TF-IDF+LogReg) **0.547** → 개선(ngram·class_weight) **0.576** (macro F1)
- 표: 모델별 F1 + Confusion Matrix. neutral 희소가 상한 요인 → 수동 검수 시 개선 여지

## 5. 토픽 / 불만 유형 분석 결과
- 토픽(비지도, TF-IDF+KMeans) **10개**: 인증·업데이트/화면·광고·UX·알림 등
- 불만 유형(룰, 키워드 사전) TOP: **오류 · UX · 인증** (슈퍼SOL 부정 리뷰 기준)
- "ML 토픽 vs 룰 유형 — 목적이 다른 두 관점" 한 줄

## 6. 경쟁 앱 벤치마킹 결과
- 표: 앱별 평균평점·부정률 (KB 4.09 / 카카오 2.63 / 우리 2.56 / **슈퍼SOL 2.55** / 하나 2.39 / 토스 2.19)
- 인사이트: 공통 최다 불만 = **오류** → 안정성 확보가 차별화. **공개 데이터 기준, 내부 전략 비추정**

## 7. RAG / LLM 리포트 생성 구조
- 입력(KPI·감성·불만·토픽·벤치마크·**근거 리뷰 Top-3**) → 프롬프트 → LLM(OpenAI) → 6섹션 리포트
- 프롬프트 원칙: **근거 없는 추정·내부 전략 추정 금지·공개 데이터 기준**
- RAG: 현재 **근거 주입형(경량)**, 벡터 인덱스(ChromaDB)는 고도화 과제 — *솔직하게 표기*
- LLM 미설정/실패 시 **룰 fallback** (생성 방식은 실제 로그 기준 표시)

## 8. 리뷰 답글 자동 생성 구조 & 안전장치
- 흐름: 대상 선택(우선순위/최신/붙여넣기) → 정책(`reply_policy.yaml`)·프롬프트 → 초안 → **안전성 재검증**
- 안전장치: 자동 게시 금지(검수용), 금지 원칙 5종, `needs_human_review`(보안/금전/심각), 금칙어 필터, 운영자 지침 우선(단 금지 원칙 우선)
- 위반 0건 (`reply_risk_cases.csv`)

## 9. Streamlit 라이브 데모
- (데모로 대체) `demo_script.md` 따라 진행: VOC → 벤치마킹 → 답글 생성(고객센터 번호 반영) → 검증
- 백업: 스크린샷 2~3장(네트워크/시간 대비)

## 10. 검증 / 오답 / 실패 케이스
- 데이터 품질 PASS, 감성 0.547→0.576, 답글 안전성 0건, 파이프라인 PASS
- 오분류: 별점↔본문 불일치(약지도 한계), 룰 오탐 → 사전 보정
- "실패를 숨기지 않고 케이스로 관리" (`error_cases.csv` / `reply_risk_cases.csv`)

## 11. 한계 & 고도화 방향
- 한계: 약지도 라벨, RSS 수집량/시점 편중, 토픽·상세 벤치마크의 전체기준 산출
- 고도화: 수동 검수 라벨, **벡터 RAG**, 임베딩 감성, 정기 자동 수집, 답글 검수 워크플로우 연동
- 클로징: "출시 초기 CX 의사결정을 **공개 데이터 + 안전한 자동화**로 지원"

### 슬라이드 작성 팁
- 한 장당 메시지 1개 + 근거 수치/그래프 1개. 텍스트 최소화.
- 약지도·룰·경량 RAG는 **있는 그대로** 표기(Q&A에서 신뢰도 ↑).
- 데모 비중을 높게(슬라이드 9가 하이라이트).
