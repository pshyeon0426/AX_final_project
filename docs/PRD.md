## 1. 배경 · 문제 정의
슈퍼SOL 출시 초기 고객 반응을 빠르게 파악하고, 경쟁 금융 앱 대비 위치를 진단하며,
부정 리뷰에 대한 응대 답글 초안을 효율적으로 만들 필요가 있다. 수작업 모니터링은
느리고 일관성이 떨어지므로, 공개 리뷰 기반의 분석·벤치마킹·답글 초안 자동화를 제공한다.

## 2. 목표 · 비목표
**목표**
- 공개 앱 리뷰로 VOC(감성·이슈·토픽)를 정량 분석
- 경쟁 앱과 평점/부정률/불만유형/기능언급 비교
- 검수용 답글 초안 생성(안전장치 포함)
- 한 화면(Streamlit)에서 수집→분석→리포트→답글까지 시연

**비목표**
- 답글 자동 게시(❌, 항상 검수용 초안)
- 실제 고객정보·내부 비공개 데이터 사용(❌)
- 경쟁사 내부 전략 추정(❌, 공개 데이터 기준만)

## 3. 사용자 · 시나리오
- **VOC/CX 담당자**: 초기 반응·주요 불만 파악, 답글 초안 활용
- **기획/전략**: 경쟁 벤치마킹으로 강·약점·개선 우선순위 도출
- 시나리오: 대시보드에서 앱·기간 선택 → KPI/차트 확인 → AI 리포트 → 부정 리뷰 답글 초안 검수

## 4. 3대 기능
1. **VOC 분석** — 감성(ML), 이슈 유형/심각도(룰), 토픽(TF-IDF+KMeans)
2. **경쟁 벤치마킹** — 평균평점·부정률·리뷰수·최근30일 변화·불만유형·기능 키워드 비교
3. **답글 초안 생성** — LLM(OpenAI) 또는 룰 fallback, 정책·안전성 검증

## 5. 아키텍처 / 데이터 흐름
```
수집(collect_reviews/app_registry) → data/raw/review_raw.csv
   └ Google Play(google-play-scraper) + App Store(iTunes RSS) + 수동/샘플
정제(preprocess) → data/processed/review_clean.csv  (+ PII [MASKED], 품질요약)
검증(validation) → data_validation_report.md / metrics.json
분석:
   ├ 감성: train_sentiment → improve_sentiment → models/sentiment_model.pkl
   ├ 이슈: issue_classifier (config/issue_keywords.yaml)
   ├ 토픽: topic_modeling
   └ 벤치마킹: benchmark
리포트: report_generator → ai_report.md
답글: reply_generator(+reply_policy.yaml, reply_prompt.md) → reply_drafts.csv → reply_eval
오케스트레이션: pipeline → metrics.json / performance_log.json
표현: app.py (Streamlit) — theme.py 공용 팔레트
```

## 6. 데이터
- **출처**: 공개 스토어 리뷰 + 검증용 샘플. 근거는 `outputs/source_manifest.csv`.
- **대상 앱(레지스트리 `config/apps.yaml`)**: 슈퍼SOL(기준) + 경쟁 5개(KB스타뱅킹·우리WON뱅킹·하나원큐·카카오뱅크·토스). ID는 Google Play 검색 + Apple iTunes 검색 API로 검증.
- **스키마**: `app_name, store, rating, date, review_text, version, source_url` (+ 정제: `clean_text, needs_human_review` 등)
- **App Store**: `app-store-scraper`가 Python 3.13 비호환이라 **iTunes 공개 RSS**로 수집(앱당 ~100~500건 상한). 부족 시 수동 CSV 보강.

## 7. 기능 상세
### 7.1 수집
- 앱 레지스트리 CRUD(대시보드 사이드바), 앱·스토어·분량 선택 → **미리보기 후 병합/덮어쓰기/버리기**, 수동 입력(CSV/직접편집).
### 7.2 전처리
- 필수 컬럼 검사, 결측·중복·초단문 제거, **PII 마스킹**(전화/이메일/계좌/주민/카드/인증번호→`[MASKED]`), 날짜 표준화, 길이 IQR 이상치, `[MASKED]` 포함 시 검수 플래그.
### 7.3 검증(합격 기준)
필수 컬럼 100% · 결측률 ≤5% · 날짜 파싱 ≥95% · 중복률 ≤3% · rating 1~5 · 총 ≥1,000건 · 슈퍼SOL+경쟁 3개↑.
### 7.4 감성(ML)
- **약지도 라벨**: 별점 1~2=neg, 3=neu, 4~5=pos (실제 감정 정답 아님 — 코드·리포트 명시).
- baseline(Dummy / TF-IDF+LogReg) → 개선(ngram·class_weight·LinearSVC). 수동검수 샘플로 별점기준/검수기준 성능 분리.
### 7.5 이슈 유형/심각도(룰)
- 10유형(로그인/인증/속도/오류/혜택/송금/투자/UX/보안/기타) 다중 라벨, 심각도 high/medium/low/positive. ML 아님 명시.
### 7.6 토픽
- TF-IDF+KMeans, 토픽당 키워드10·대표리뷰5, `topic_label` 검수 컬럼, 짧은 리뷰 필터로 쏠림 완화.
### 7.7 벤치마킹
- 앱별 핵심 지표 + 불만유형 TOP5 + 기능 키워드 언급률 + 강점/약점/개선기회. **공개 데이터 기준, 내부 전략 비추정**.
### 7.8 AI 리포트
- 6섹션(핵심요약/부정원인/개선우선순위/벤치마킹/근거/한계). 프롬프트에 "근거 없는 추정·내부 전략 추정 금지·공개 데이터 기준" 명시. LLM 실패/미설정 시 룰 fallback. 대시보드에서 **현재 필터 기준 재생성** 지원.
### 7.9 답글 초안 (§9 정책 참조)
- 대상: 기존 리뷰(우선순위순/최신순) 또는 **직접 붙여넣기**. 운영자 🧭 지침 우선 반영. 생성 후 안전성 재검증.

## 8. 대시보드(app.py) 요건
- 사이드바 4그룹: ①데이터 소스(수집/업로드/샘플) ②분석 필터(앱·경쟁·기간, 종료 기본=오늘) ③실행(파이프라인/새로고침) ④상태(OpenAI).
- 메인: 컨텍스트 배지+파이프라인 상태, KPI 카드, 차트(추이·감성·불만유형·앱별비교), 부정 리뷰.
- 탭: AI 리포트 · 경쟁 벤치마킹 · 토픽 · 답글 생성 · 검증 · 원본 리뷰(서브).
- **필터 반영**: KPI·차트·불만유형·앱별비교·벤치마킹 요약은 선택 기간 실시간 반영. 토픽/상세 벤치마크 지표는 파이프라인 산출물(전체 기준, 화면에 명시).
- **표시**: 영문 컬럼/값 한글 매핑, 어려운 항목 도움말(hover), 날짜는 날짜만.
- **상태 SSOT**: LLM/룰 표시는 `performance_log` 실제 기록 기준. 캐시는 파일 지문으로 무효화.
- **30초 기준**: "사전 학습 모델/인덱스 준비 후 분석 요청" 기준임을 화면에 안내.
- **테마**: `src/theme.py` + `.streamlit/config.toml` (브랜드 블루 #2F6FE0).

## 9. 답글 정책 · 안전장치 (`config/reply_policy.yaml`)
- 자동 게시 금지(검수용 초안). 답글 유형 6종, 한국어 존댓말 2~4문장.
- 금지 원칙 5종(개인정보 요청·확정 약속·거래내역 추정·투자 조언·경쟁사 비방).
- 검수 필수: 심각도 high, 보안/송금/오류, 민감 키워드.
- 운영자 지침(🧭) 우선 반영(금지 원칙 충돌 시 금지 원칙 우선), 모델은 사실 미창작 → 실제 값 입력 필요.
- 고객센터 안내 문구/번호는 정책 파일에서 관리(플레이스홀더, 운영팀 확인 후 교체).
- 후처리 `safety_check_reply` 로 위반 재검증, 위반은 `reply_risk_cases.csv` 기록.

## 12. 알려진 한계 / 향후 과제
- 감성 라벨이 별점 기반 약지도 → 실제 감정과 불일치 가능. 수동 검수 라벨 확보 시 성능·신뢰도 향상.
- App Store RSS 수집량 상한, 일부 앱 스토어 데이터 편중(특정 월 쏠림).
- 토픽/상세 벤치마크는 파이프라인 산출물(전체 데이터) 기준 — 실시간 기간 재계산 미적용.
- 정기 자동 수집(스케줄러), RAG 인덱스 기반 근거 강화, 임베딩 감성 모델은 운영 단계 과제.
- 품질 점검 체크리스트: `docs/qa_scenario.md`.
