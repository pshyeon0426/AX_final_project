# 예상 Q&A — 슈퍼SOL 고객 반응 분석

발표 후 질의응답 대비. 각 답변은 **핵심 1줄 + 근거**로 구성.

## Q1. 왜 이 주제를 선택했는가?
- 슈퍼SOL **출시 초기**에는 고객 반응을 빠르게 읽고 경쟁 앱 대비 위치를 파악하는 것이 중요한데,
  수작업 모니터링은 느리고 일관성이 떨어집니다.
- 공개 앱 리뷰는 **즉시 확보 가능한 1차 VOC**이며, 분석·벤치마킹·답글까지 한 흐름으로 자동화하면
  CX/기획 실무에 바로 도움이 됩니다. 내부 데이터 없이도 가능한 점이 매력적이었습니다.

## Q2. 머신러닝은 어디에 썼는가?
- **감성 분류**(지도학습): TF-IDF + LogisticRegression/LinearSVC (`train_sentiment`/`improve_sentiment`)
- **토픽 모델링**(비지도): TF-IDF + KMeans (`topic_modeling`)
- 반면 **이슈 유형/심각도 분류는 ML이 아니라 키워드 사전 기반 룰**입니다(MVP 속도·해석성 우선,
  `config/issue_keywords.yaml`). "어디는 ML, 어디는 룰"을 의도적으로 구분했습니다.

## Q3. F1-score는 어떤 라벨 기준인가?
- **별점 기반 약지도(weak supervision) 라벨**입니다: 1~2점=부정, 3점=중립, 4~5점=긍정.
- 작성자의 **실제 감정 정답이 아니므로** 성능은 "별점 라벨 재현도"로 해석해야 합니다.
  이 점을 코드 주석·리포트·발표에 명시했습니다. 향후 수동 검수 라벨로 검증이 필요합니다.

## Q4. baseline 대비 어떻게 개선했는가?
- baseline: TF-IDF **unigram** + LogReg(기본값), macro F1 **0.547** (Dummy 0.23 대비 향상).
- 개선: **(1,2)~(1,3)gram**, **class_weight='balanced'**, max_features 조정, LinearSVC 비교 →
  최종 **0.576**. neutral 클래스가 희소(소수)해 상한이 제약되며, 이는 약지도 라벨의 구조적 한계입니다.
- 데모상 **수동 검수 기준**으로 평가하면 점수가 더 오르는 것을 확인 → 모델보다 **라벨 품질**이 관건.

## Q5. RAG가 왜 필요한가?
- 리포트가 **근거 없이 그럴듯한 말**을 하지 않도록, 실제 리뷰를 **근거로 주입**해 생성하기 위함입니다.
- 현재는 **근거 리뷰 Top-3를 프롬프트에 주입하는 경량 방식**이고, 대규모 리뷰에서 의미 검색으로
  근거를 뽑는 **벡터 RAG(ChromaDB)** 는 고도화 과제입니다(의존성은 준비). 과장 없이 현 수준을 표기했습니다.

## Q6. 답글 자동 생성은 안전한가?
- 다층 안전장치: ① **자동 게시 금지**(검수용 초안) ② 금지 원칙 5종(개인정보 요청·확정 약속·거래내역
  추정·투자 조언·경쟁사 비방) ③ 보안/금전/심각 이슈는 **`needs_human_review=True`** 자동 표시
  ④ 생성 후 `safety_check_reply` 로 금칙어·민감정보 요청·확정 약속·투자 조언 **재검증**.
- 결과: 평가 샘플에서 안전성 위반 **0건**(`reply_risk_cases.csv`). 모델은 번호 등 사실을 지어내지 않습니다.

## Q7. OpenAI API 키는 어떻게 관리했는가?
- 키는 **코드에 하드코딩하지 않고 `.env`에만** 두며, `.gitignore`로 커밋을 차단합니다(`.env.example`만 공유).
- 모델명은 `OPENAI_MODEL` 환경변수로 분리해 교체 가능. 키 미설정 시 **룰 fallback**으로 동작합니다.
- 채팅/문서/화면 공유에 키를 노출하지 않으며, **노출되면 즉시 폐기·재발급**하는 것을 원칙으로 합니다.

## Q8. 실제 고객 응대에 바로 자동 게시해도 되는가?
- **아니요.** 본 시스템의 답글은 **담당자 검수용 초안**이며, 시스템이 스토어에 직접 게시하지 않습니다.
- 금융 도메인 특성상 보상·장애 원인·처리 결과를 확정 약속할 수 없고, 민감 이슈는 사람이 판단해야 하므로
  **사람 검수(human-in-the-loop)** 를 전제로 설계했습니다.

## Q9. 앱스토어 리뷰의 대표성 한계는 무엇인가?
- 리뷰 작성자는 **불만/만족이 강한 일부**에 치우치는 자기선택 편향이 있어 전체 사용자를 대표하지 않습니다.
- 또 **App Store는 공개 RSS 수집량 상한**(앱당 ~100~500건)이 있고, 수집 시점에 따라 특정 기간에 편중될 수
  있습니다. 따라서 절대 수치보다 **상대 비교·추세·유형 분포**로 해석하고, 필요 시 설문 등으로 보완해야 합니다.

## Q10. 30초 성능 기준은 무엇인가?
- "**사전 학습된 모델과 사전 생성된 인덱스/산출물이 준비된 상태에서, 분석을 요청할 때**"의 응답 기준입니다.
- **최초 학습·수집·인덱싱 시간은 별도**이며, 화면에도 이 기준을 명시했습니다. 대시보드의 KPI/차트는
  사전 산출물을 읽어 즉시 표시되고, LLM 답글 생성만 호출당 수 초가 소요됩니다.

## Q11. 내부 데이터 없이 실무 활용 가능성이 있는가?
- 가능합니다. **공개 리뷰만으로** 초기 반응 진단·경쟁 벤치마킹·답글 초안까지 제공하므로,
  내부 데이터 연동 전에도 CX/기획이 바로 활용할 수 있습니다.
- 내부 데이터(거래·문의 로그 등)와 결합하면 정확도·범위가 확장되지만, 이는 **선택적 고도화**이지 전제 조건이 아닙니다.
